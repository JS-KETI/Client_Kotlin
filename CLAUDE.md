# 1. 프로젝트 개요

**Client_Kotlin**은 MoQ(Media over QUIC) 기반 드론·무인이동체 publisher의 **안드로이드 버전** 신규 구현 프로젝트다. 기존 Jetson + Rust 드론(`../Client_Dron/`)이 수행하던 publisher 역할을 안드로이드 단말로 이식하되, **서버·관제·moq-relay는 전부 무수정 재사용**한다.

## 핵심 요구사항

- 카메라 캡처 → H.264 인코딩 → MoQ broadcast publish
- 다중 단말 동시 publish (서버 broadcast 모델로 자동 지원)
- 실시간 데이터 전송량 표시 + 서버 보고 (3초 간격)
- **Wi-Fi ↔ Cellular 무중단 네트워크 전환** (QUIC connection migration 활용)

- **환경 파일**: `.claude/.env`
- **API Key**: `GEMINI_API_KEY` 변수

```bash
GEMINI_API_KEY=$(grep '^GEMINI_API_KEY=' .claude/.env | cut -d'=' -f2)
GEMINI_MODEL=$(grep '^GEMINI_MODEL=' .claude/.env | cut -d'=' -f2)

curl -s "https://generativelanguage.googleapis.com/v1beta/models/${GEMINI_MODEL}:generateContent?key=${GEMINI_API_KEY}" \
  -H "Content-Type: application/json" \
  -d '{
    "contents": [{
      "parts": [{
        "text": "[프롬프트 내용]"
      }]
    }]
  }'
```

응답 추출: `candidates[0].content.parts[0].text`

## Codex CLI

- **환경 파일**: `.claude/.env`
- **CLI 경로**: `CODEX_CLI_PATH` 변수
- **필수 플래그**: `--skip-git-repo-check`


# 9. Repository (Android Data Layer) 패턴

myYak 서버의 `Service (Interface + Impl)` 패턴과 동일하게, Android Repository도 인터페이스 + Impl 분리.

```kotlin
// data/rest/DeviceRepository.kt
interface DeviceRepository {
    suspend fun register(request: DeviceRegisterRequest): Result<DeviceSummary>
    suspend fun updateTelemetry(deviceId: String, request: DeviceTelemetryRequest): Result<Unit>
}

// data/rest/DeviceRepositoryImpl.kt
class DeviceRepositoryImpl(
    private val api: DeviceApi
) : DeviceRepository {

    override suspend fun register(request: DeviceRegisterRequest): Result<DeviceSummary> =
        runCatching {
            api.register(request).data ?: error("empty response")
        }

    override suspend fun updateTelemetry(
        deviceId: String,
        request: DeviceTelemetryRequest
    ): Result<Unit> = runCatching {
        api.updateTelemetry(deviceId, request)
    }
}
```

## ViewModel ↔ UseCase ↔ Repository

```kotlin
class PublisherViewModel(
    private val connectUseCase: ConnectUseCase,
    private val switchNetworkUseCase: SwitchNetworkUseCase
) : ViewModel() {

    fun onConnect() = viewModelScope.launch {
        connectUseCase().onFailure { reportError(it) }
    }
}

class ConnectUseCase(
    private val deviceRepository: DeviceRepository,
    private val moqPublisher: MoqPublisher
) {
    suspend operator fun invoke(): Result<Unit> = runCatching {
        val summary = deviceRepository.register(buildRegisterRequest()).getOrThrow()
        moqPublisher.connect(summary.relayUrl, summary.broadcastPath)
    }
}
```

# 11. API 설계 원칙

> 본 저장소는 **클라이언트 측**이므로 서버 API를 *호출*만 한다. 서버 측 API 설계 변경은 불가하다. 다만 클라이언트가 지켜야 할 호출 원칙은 다음과 같다.

## 11-1. 일괄 처리(Batch) 원칙

서버 API가 일괄 처리를 지원하지 않더라도, 클라이언트에서 동일 API를 짧은 시간에 반복 호출하지 않는다.

```kotlin
❌ 잘못된 예 — 매 프레임마다 텔레메트리 호출
launch {
    captureLoop.collect { frame ->
        deviceRepository.updateTelemetry(...)  // 초당 30번 호출 = 서버 부담
    }
}

✅ 올바른 예 — 별도 주기 타이머
launch {
    while (isActive) {
        delay(3_000)
        deviceRepository.updateTelemetry(buildTelemetry())  // 3초마다 1회
    }
}
```

## 11-2. 서버 컨트랙트 위반 금지

| 항목 | 강제 사항 |
|---|---|
| broadcast 이름 형식 | `{deviceId}/{streamId}` 정확히 |
| streamId 기본값 | `"main"` |
| relayUrl | 서버 응답의 값을 그대로 사용 (포트·namespace 수정 금지) |
| codec 문자열 | `avc1.<6 hex>` (SPS의 profile/level에서 추출) |
| 텔레메트리 주기 | 3초 (Rust 드론과 동일) |
| 시간 단위 | timestamp는 microsecond (`MediaCodec.BufferInfo.presentationTimeUs`) |

# 12. Design Principles (SOLID)

> 모든 코드 작성 및 리팩토링 시 아래 5가지 SOLID 원칙을 준수한다.

## 1. SRP (Single Responsibility Principle)
- ViewModel은 UI 상태만, UseCase는 비즈니스 로직만, Repository는 데이터 접근만
- `CameraEncoder`가 `MediaCodec` 설정과 NAL 추출을 모두 책임지지 않도록 분리

## 2. OCP (Open-Closed Principle)
- 새 코덱 추가 시 기존 `MoqPublisher` 수정 없이 `CatalogBuilder` 구현체 확장으로 대응

## 3. LSP (Liskov Substitution Principle)
- `MoqPublisher` interface를 구현하는 어떤 구현체든 동일한 계약(connect/publish/finish) 유지

## 4. ISP (Interface Segregation Principle)
- `DeviceRepository`에 텔레메트리·등록·조회를 모두 묶지 말고 필요 시 분리

## 5. DIP (Dependency Inversion Principle)
- ViewModel은 `DeviceRepository` interface에 의존, 구현체 직접 의존 금지
- Compose에서 `MoqClient` 직접 호출 금지 — UseCase/Repository 거치도록

# 13. Adapter Pattern (교체 가능한 모듈)

> 외부 라이브러리·플랫폼 API처럼 **교체될 가능성이 있는 모듈**에는 어댑터 패턴을 적용한다.

## 적용 대상

- **MoQ 클라이언트**: `dev.moq:moq` → 향후 다른 MoQ 구현체로 교체 가능성 대비
- **카메라**: CameraX → Camera2 fallback
- **REST 클라이언트**: Retrofit → Ktor (선택)
- **인증 토큰 저장**: SharedPreferences → EncryptedSharedPreferences / DataStore

## 폴더 구조 (제안)

```
data/moq/
├── MoqPublisher.kt              # 인터페이스 (Port)
├── MoqPublisherImpl.kt          # dev.moq:moq 기반 구현체 (Adapter)
└── CatalogBuilder.kt            # 코덱별 catalog 생성 로직
```

## 구현 패턴

```kotlin
// Port
interface MoqPublisher {
    suspend fun connect(relayUrl: String, broadcastPath: String): Result<Unit>
    suspend fun publishMedia(codec: String, initBytes: ByteArray)
    suspend fun writeFrame(payload: ByteArray, timestampUs: Long)
    suspend fun finish()
}

// Adapter (dev.moq:moq 구현체)
class MoqPublisherImpl : MoqPublisher {
    private val client = MoqClient()
    private var session: MoqSession? = null
    private var broadcast: MoqBroadcastProducer? = null

    override suspend fun connect(...) = runCatching { ... }
}
```

ViewModel/UseCase는 `MoqPublisher` 인터페이스에만 의존.
